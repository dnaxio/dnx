# dnx.yaml — Complete Reference

The `dnx.yaml` file is the single entry point for your deployment configuration.

## Global Structure

```yaml
version: "1"
name: "my-project"

environments:
  staging: { ... }
  production: { ... }

workloads:
  - name: web
    ...

proxy:
  routes:
    - domain: app.example.com

health:
  interval: 30s

preview:
  enabled: true
```

## `environments`

```yaml
environments:
  staging:
    servers:
      - name: staging-1
        host: 10.0.1.10
        port: 22
        user: deploy
        key_path: ~/.ssh/id_ed25519
    deploy_strategy: rolling
    auto_rollback: true
    health_check_grace: 30s

  production:
    servers:
      - name: prod-1
        host: 10.0.10.1
      - name: prod-2
        host: 10.0.10.2
    deploy_strategy: blue-green
```

## `workloads`

All workloads are defined in a single unified list. Each workload can use either flox (source-based) or an OCI image (image-based).

```yaml
workloads:
  - name: web
    driver: flox                     # flox | docker
    start_cmd: "bun run server.ts"   # optional (flox on-activate handles it)
    build:
      local:
        steps:
          - run: bun generate
          - run: bun run build
      server:
        steps:
          - run: bun install
          - run: bun run build
    ports: [3000]                    # optional (only if exposing ports)
    image: ghcr.io/org/web:v1        # optional — OCI image (replaces flox build)
    workdir: /app                   # default: /app
    volumes:                         # extra Docker volumes
      - /data:/data
    restart: "no"                    # no | always | on-failure | unless-stopped
    env:
      NODE_ENV: "{{ .Environment }}"
      DATABASE_URL: "${secret:DATABASE_URL}"
    sync:
      source: ./
      force: true                   # force sync (skip hash check)
      include:                       # force inclusion even if in .gitignore
        - .output
        - dist
      exclude: [tests]               # default: .dnax node_modules .git .flox .cursor
```

| Field | Required | Description |
|-------|----------|-------------|
| `name` | Yes | Unique workload name (lowercase, alphanumeric and hyphens only) |
| `driver` | Yes | Runtime driver: `flox`, `docker` |
| `start_cmd` | No | Command to start the process (optional with flox on-activate) |
| `ports` | No | Ports to expose (optional — omit for internal workloads) |
| `image` | No | OCI container image reference (e.g., `ghcr.io/org/web:v1`). When set, skips source build and pulls the image instead |
| `workdir` | No | Working directory (default: `/app`) |
| `volumes` | No | Extra Docker volumes |
| `restart` | No | Restart policy: `no`, `always`, `on-failure`, `unless-stopped` |
| `env` | No | Environment variables |
| `build` | No | Build steps (local + server phases) |
| `sync` | No | Source sync configuration |

### Sync (`sync.source`, `sync.exclude`, `sync.include`, `sync.force`)

```yaml
sync:
  source: ./
  force: true
  include:
    - .output
    - dist
  exclude:
    - tests
```

| Field | Required | Description |
|-------|----------|-------------|
| `source` | No | Source directory to sync (default: `.`) |
| `force` | No | Force sync (skip hash change check) |
| `include` | No | Force inclusion of files/directories even if in `.gitignore` or default exclusions |
| `exclude` | No | Additional exclusions (added to defaults and `.gitignore`) |

### Build (`build.local` and `build.server`)

```yaml
build:
  local:
    steps:
      - run: bun generate
      - run: bun build
  server:
    steps:
      - run: bun install
      - run: bun run build
```

| Phase | Where | When |
|-------|-------|------|
| `build.local` | Host machine | `dnx build` or automatically before `dnx deploy` |
| `build.server` | Remote servers | During `dnx deploy`, after sync |

Both phases are optional. If a phase has no steps, it is simply skipped.

## `proxy` (Caddy)

```yaml
proxy:
  auto_ssl: true
  email: admin@example.com
  routes:
    - domain: app.example.com
      target: web
      port: 3000
      lb_policy: round_robin
      ssl: true
```

## Variables & Templates

### `.env` Files (automatic loading)

DNX automatically loads `.env` files from your project root. **No configuration needed.**

| File | Priority | Purpose |
|------|----------|---------|
| `.env` | 1 (lowest) | Shared variables |
| `.env.<environment>` | 2 | e.g., `.env.production`, `.env.staging` |
| `.dnax/.env` | 3 | DNX-specific variables |
| `.dnax/.env.<environment>` | 4 | e.g., `.dnax/.env.production` |
| `.env.local` | 5 (highest) | Local overrides (never commit) |
| Shell (`export`) | 6 (max) | Always wins |

Example:

```bash
# .env
PROD_HOST=prod.example.com
STAGING_HOST=staging.example.com

# .env.production (overrides .env)
PROD_HOST=prod2.example.com

# .env.local (overrides everything except shell)
PROD_HOST=localhost
```

When you run `dnx deploy web --env production --tag v1`, DNX loads `.env` → `.env.production` → `.env.local`. Shell variables (`export PROD_HOST=...`) are never overwritten.

To deploy all workloads: `dnx deploy --all --env production --tag v1`.

### In `dnx.yaml`

Inside `dnx.yaml`, you can use:

| Syntax | Resolution |
|--------|-----------|
| `${ENV_VAR}` | Environment variable |
| `${ENV_VAR:-default}` | With fallback |
| `${secret:KEY}` | Encrypted secret (decrypted at deploy) |
| `{{ .Environment }}` | Current environment name |
