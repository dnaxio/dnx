# Secrets

DNX chiffre vos secrets avec AES-256-GCM et les injecte automatiquement au moment du déploiement.

## Initialisation

```bash
dnx secrets init
```

Cela génère une clé maîtresse AES-256 dans `~/.dnx/master.key` (permissions 600).

## Définir un secret

```bash
dnx secrets set DATABASE_URL=postgres://user:pass@host:5432/db --env production
dnx secrets set REDIS_URL=redis://localhost:6379 --env all
```

Le flag `--env` permet de restreindre un secret à un environnement spécifique.

## Lister les secrets

```bash
dnx secrets print
# DATABASE_URL: [production]  2026-06-07 10:00:00
# REDIS_URL:    [all]         2026-06-07 10:00:00
```

Avec `--json` pour une sortie machine-readable.

## Extraire les secrets

```bash
# Format env (défaut)
dnx secrets extract --env production
# DATABASE_URL=postgres://...
# REDIS_URL=redis://...

# Format JSON
dnx secrets extract --env production --format json
```

## Supprimer un secret

```bash
dnx secrets remove DATABASE_URL --env production
```

## Utilisation dans dnx.yaml

Dans votre configuration, référencez un secret avec `${secret:KEY}` :

```yaml
apps:
  - name: web
    runtime:
      env:
        DATABASE_URL: "${secret:DATABASE_URL}"
```

Au moment du déploiement, `${secret:DATABASE_URL}` est remplacé par la valeur déchiffrée.

## Sécurité

- Clé maîtresse : `~/.dnx/master.key` (chmod 600)
- Alternative : variable d'environnement `DNX_MASTER_KEY`
- **Ne jamais commiter `master.key` dans Git**
- Les secrets sont chiffrés au repos dans SQLite (`~/.dnx/state.db`)
