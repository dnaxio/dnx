# Environnements

DNX supporte trois environnements : `staging`, `test` et `production`.

## Définition

```yaml
environments:
  staging:
    servers:
      - name: staging-1
        host: 10.0.1.10
    deploy_strategy: rolling
    auto_rollback: true

  production:
    servers:
      - name: prod-1
        host: 10.0.10.1
      - name: prod-2
        host: 10.0.10.2
    deploy_strategy: blue-green
    auto_rollback: true
    health_check_grace: 60s
```

## Fichiers d'override par environnement

Vous pouvez créer des fichiers séparés pour surcharger la configuration par environnement :

```
dnx.yaml           # Configuration de base
dnx.staging.yaml   # Overrides pour staging
dnx.production.yaml # Overrides pour production
```

## Fichiers `.env` par environnement

DNX charge automatiquement le fichier `.env.<environnement>` correspondant :

```bash
# .env.production — chargé automatiquement avec --env production
PROD_HOST=prod.example.com
DB_URL=postgres://prod-db:5432

# .env.staging — chargé automatiquement avec --env staging
PROD_HOST=staging.example.com
DB_URL=postgres://staging-db:5432
```

Aucune configuration manuelle nécessaire — si vous déployez en `production`, `.env.production` est chargé automatiquement.

Exemple `dnx.staging.yaml` :

```yaml
environments:
  staging:
    proxy:
      routes:
        - domain: staging.app.example.com
          target: web
          port: 3000
```

## Stratégies de déploiement

| Stratégie | Description | Use case |
|-----------|------------|----------|
| `rolling` | Déploie serveur par serveur | Staging, petits déploiements |
| `blue-green` | Nouvel ensemble → switch → ancien détruit | Production, zero-downtime |
| `canary` | Déploie sur un sous-ensemble, vérifie, propage | Production, déploiements risqués |

## Diff entre environnements

```bash
dnx config diff staging production
```

Affiche les différences entre deux environnements (serveurs, stratégie, variables, etc.).
