# dnx.yaml — Référence complète

Le fichier `dnx.yaml` est le point d'entrée unique de votre configuration de déploiement.

## Structure globale

```yaml
version: "1"
name: "my-project"

environments:       # Définition des environnements (staging, test, production)
  staging: { ... }
  production: { ... }

workloads:          # Liste des workloads (apps + services unifiés)
  - name: web
    driver: flox
    ...

  - name: redis
    image: redis:7-alpine
    ...

proxy:              # Configuration du reverse proxy Caddy
  routes:
    - domain: app.example.com
      ...

health:             # Health checks globaux
  interval: 30s

preview:            # Preview/Ephemeral environments
  enabled: true

notifications:      # Notifications de déploiement
  deploy_success:
    - type: webhook
```

## `environments`

```yaml
environments:
  staging:
    servers:
      - host: $STAGING_HOST
        user: $USER
        password: $SSH_PASS      # optionnel
        key_path: ~/.ssh/id_ed25519  # optionnel
    deploy_strategy: rolling       # rolling | blue-green | canary
    auto_rollback: true
    health_check_grace: 30s
    min_healthy_percent: 75

  production:
    servers:
      - host: 10.0.10.1
        user: root
      - host: 10.0.10.2
        user: root
    deploy_strategy: blue-green
```

## `workloads`

Un workload peut être soit du code source buildé via flox/devbox/docker, soit une image OCI pré-buildée (Redis, Postgres...). Tout est unifié sous `workloads`.

### Workload from source (flox)

```yaml
workloads:
  - name: web
    driver: flox                     # flox | docker | devbox | podman
    start_cmd: "bun run server.ts"   # optionnel
    build:
      local:
        steps:
          - run: bun generate
          - run: bun run build
      server:
        steps:
          - run: bun install
          - run: bun run build
    ports: [3000]                    # optionnel (défaut: [])
    workdir: /app                   # défaut: /app
    volumes:                         # volumes Docker additionnels
      - /data:/data
    restart: "no"                    # no | always | on-failure | unless-stopped
    env:
      NODE_ENV: "{{ .Environment }}"

    # Pour driver: docker
    dockerfile: ./Dockerfile
    registry: ghcr.io/org/web

    sync:                            # Fichiers à envoyer au serveur
      source: ./
      force: true                    # force le sync (ignore le hash)
      include:                       # force l'inclusion même si dans .gitignore
        - .output
        - dist
      exclude:                       # défaut: .dnax node_modules .git .flox .cursor
        - tests

    health:                          # Optionnel
      type: http
      endpoint: /health
      interval: 30s
      timeout: 5s
      retries: 3

    scaling:                         # Optionnel
      min_instances: 2
      max_instances: 10
      target_cpu: 70
      target_memory: 80
      cooldown: 300s
```

### Workload from image (OCI)

```yaml
workloads:
  - name: postgres
    image: postgres:16
    ports: [5432]
    volumes:
      - pgdata:/var/lib/postgresql/data
    env:
      POSTGRES_PASSWORD: "${secret:DB_PASS}"
    restart: always

  - name: redis
    image: redis:7-alpine
    ports: [6379]
    volumes:
      - redis-data:/data
```

### Build (`build.local` et `build.server`)

```yaml
build:
  local:
    steps:
      - run: bun generate
      - run: bun build
  server:
    steps:
      - run: bun install
      - run: bun run build
```

| Phase | Où | Quand |
|-------|-----|-------|
| `build.local` | Machine hôte | `dnx build` ou automatiquement avant `dnx deploy` |
| `build.server` | Serveur distant | Pendant `dnx deploy`, après le sync |

Les deux phases sont optionnelles. Si une phase n'a pas d'étapes, elle est simplement ignorée.

### Champs du workload

| Champ | Type | Défaut | Description |
|-------|------|--------|-------------|
| `name` | string | — | Nom unique du workload (minuscules, alphanumérique + tirets uniquement) |
| `image` | string | — | Image OCI à pull (optionnel, si absent → build from source) |
| `driver` | enum | `flox` | flox \| docker \| devbox \| podman |
| `start_cmd` | string | — | Commande de démarrage |
| `ports` | int[] | `[]` | Ports exposés (optionnel) |
| `workdir` | string | `/app` | Répertoire de travail |
| `volumes` | string[] | `[]` | Volumes Docker |
| `restart` | enum | `no` | no \| always \| on-failure \| unless-stopped |
| `env` | object | `{}` | Variables d'environnement |
| `flox_env` | string | — | Chemin vers l'env flox |
| `dockerfile` | string | — | Dockerfile (driver: docker) |
| `registry` | string | — | Registry OCI |
| `build` | object | — | Étapes de build (local + server) |
| `health` | object | — | Health check |
| `scaling` | object | — | Auto-scaling |
| `sync` | object | — | Sync des fichiers |

### Sync (`sync.source`, `sync.exclude`, `sync.include`, `sync.force`)

```yaml
sync:
  source: ./
  force: true
  include:
    - .output
    - dist
  exclude:
    - tests
```

| Champ | Type | Défaut | Description |
|-------|------|--------|-------------|
| `source` | string | `.` | Dossier source à synchroniser |
| `force` | boolean | `false` | Force le sync (ignore le hash de changement) |
| `include` | string[] | `[]` | Force l'inclusion de fichiers/dossiers même s'ils sont dans `.gitignore` ou les exclusions par défaut |
| `exclude` | string[] | `[]` | Exclusions supplémentaires (ajoutées aux défauts et `.gitignore`) |

## `proxy` (Caddy)

```yaml
proxy:
  auto_ssl: true
  email: admin@example.com
  routes:
    - domain: app.example.com
      target: web
      port: 3000
      lb_policy: round_robin     # round_robin | least_conn | first | ip_hash
      ssl: true
      headers:
        X-Frame-Options: DENY
        X-Content-Type-Options: nosniff

  global_options:
    rate_limit:
      rate: 100
      burst: 50
      period: 1s
```

## `health`

```yaml
health:
  enabled: true
  interval: 30s
  alerts:
    - type: webhook
      url: https://hooks.slack.com/xxx
      on: [degraded, unhealthy, recovered]
    - type: email
      to: ops@example.com
      on: [unhealthy]
```

## `preview`

```yaml
preview:
  enabled: true
  domain_template: "preview-{{ .BranchSlug }}.example.com"
  ttl_hours: 48
  max_concurrent: 5
  auto_deploy:
    on_push: true
    branches: ["feat/*", "fix/*"]
  notifications:
    github:
      comment_pr: true
      status_check: true
```

## Variables et templates

### Fichiers `.env` (chargement automatique)

DNX charge automatiquement les fichiers `.env` à la racine du projet. **Aucune configuration nécessaire.**

| Fichier | Priorité | Rôle |
|---------|----------|------|
| `.env` | 1 (faible) | Variables partagées |
| `.env.<environnement>` | 2 | Ex: `.env.production`, `.env.staging` |
| `.dnax/.env` | 3 | Variables spécifiques à DNX |
| `.dnax/.env.<environnement>` | 4 | Ex: `.dnax/.env.production` |
| `.env.local` | 5 (fort) | Overrides locaux (jamais commité) |
| Shell (`export`) | 6 (max) | Priorité absolue |

Exemple :

```bash
# .env
PROD_HOST=prod.example.com
STAGING_HOST=staging.example.com

# .env.production (surcharge .env)
PROD_HOST=prod2.example.com

# .env.local (surcharge tout, sauf le shell)
PROD_HOST=localhost
```

Quand vous lancez `dnx deploy web --env production --tag v1`, DNX charge `.env` → `.env.production` → `.env.local`. Les variables du shell (`export PROD_HOST=...`) ne sont jamais écrasées.

Pour déployer tous les workloads : `dnx deploy --all --env production --tag v1`.

### Dans le `dnx.yaml`

Dans le `dnx.yaml`, vous pouvez utiliser :

| Syntaxe | Résolution |
|---------|-----------|
| `${ENV_VAR}` | Variable d'environnement |
| `${ENV_VAR:-default}` | Avec fallback |
| `${secret:KEY}` | Secret chiffré (déchiffré au déploiement) |
| `{{ .Environment }}` | Nom de l'environnement courant |
