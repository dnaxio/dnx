# Cycle de vie

## Workloads

Tous les workloads (API, workers, Redis, Postgres...) sont gérés avec la même commande `dnx workload`.

```bash
# Démarrer
dnx workload start web --env staging

# Arrêter (graceful: SIGTERM → wait 30s → SIGKILL)
dnx workload stop web

# Redémarrer
dnx workload restart web

# Statut (PID, uptime, CPU, RAM, ports)
dnx workload status web --env production

# Logs
dnx workload logs web --follow --tail 200 --grep "error"

# Supprimer (avec confirmation)
dnx workload remove web --env staging --force
```

## Logs

```bash
# Dernières 100 lignes
dnx workload logs web --env staging

# Suivre en continu
dnx workload logs web --follow

# Filtrer par motif
dnx workload logs web --grep "ERROR"

# Serveur spécifique
dnx workload logs web --server prod-1
```

## Statut détaillé

```bash
dnx workload status web --env production
# 🟢 prod-1: RUNNING PID=12345 UPTIME=3600
# 🟢 prod-2: RUNNING PID=23456 UPTIME=3600
# 🔴 prod-3: STOPPED
```

## Workloads OCI (image pré-buildée)

Pour les workloads basés sur une image (Redis, Postgres, etc.), le fonctionnement est identique :

```bash
dnx workload start redis --env staging
dnx workload status redis --env production
dnx workload logs redis --follow
dnx workload remove redis --force
```

DNX utilise `docker run` / `docker start` pour les workloads avec `image:`.
