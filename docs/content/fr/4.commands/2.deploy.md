# dnx deploy

```bash
dnx deploy <workload> [options]
```

## Options

| Option | Description | Défaut |
|--------|-------------|--------|
| `--env <env>` | Environnement cible | `staging` |
| `--strategy <s>` | rolling, blue-green, canary | Configuré dans dnx.yaml |
| `--dry-run` | Simule sans exécuter | `false` |
| `--no-health-check` | Désactive le health check | `false` |
| `--timeout <s>` | Timeout SSH (secondes) | `30` |

## Exemples

```bash
# Déploiement simple en staging (rolling)
dnx deploy web --env staging

# Production en blue-green
dnx deploy web --env production --strategy blue-green

# Dry-run (vérification sans déployer)
dnx deploy web --env production --dry-run

# Déploiement avec timeout custom
dnx deploy api --env production --timeout 60

# Déployer tous les workloads
dnx deploy --all --env production --tag v1.2.3
```

## Cycle de déploiement

1. **Build local** — exécute `build.local.steps` sur la machine hôte (si configuré)
2. **Vérification des prérequis** — flox, caddy installés sur les serveurs
3. **Acquisition du lock** — empêche les déploiements concurrents
4. **Sync** — synchronise les sources vers les serveurs
5. **Build server** — exécute `build.server.steps` sur les serveurs distants (si configuré)
6. **Release** — crée la nouvelle release (`$HOME/.dnx/workloads/<name>/releases/<version>`)
7. **Start** — démarre la nouvelle version
8. **Health check** — vérifie que le workload répond
9. **Proxy** — met à jour Caddy
10. **Release du lock**

## Stratégies

### Rolling

Déploie séquentiellement, serveur par serveur. Une partie du trafic continue sur l'ancienne version.

```bash
dnx deploy web --strategy rolling
```

### Blue-Green

Déploie sur un nouvel ensemble de serveurs, puis switch le proxy d'un coup.

```bash
dnx deploy web --strategy blue-green
```

### Canary

Déploie sur un sous-ensemble (ex: 1 serveur sur 5), vérifie, puis propage.

```bash
dnx deploy web --strategy canary
```
