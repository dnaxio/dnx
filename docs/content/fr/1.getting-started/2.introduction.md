# Introduction

DNX Deploy est un outil de déploiement multi-serveurs basé sur SSH, conçu pour les équipes qui veulent une alternative simple à Docker Swarm, Kubernetes ou Capistrano.

## Pourquoi DNX ?

| Besoin | DNX | Docker Swarm | Kubernetes | Capistrano |
|--------|-----|-------------|------------|------------|
| Simple (un binaire) | ✅ | ❌ | ❌ | ❌ (Ruby) |
| Multi-serveur SSH | ✅ 50 serveurs | ❌ | ✅ | ✅ |
| Sans agent (agentless) | ✅ | ❌ | ❌ | ✅ |
| Envs reproductibles | ✅ flox + OCI | ✅ Docker | ✅ OCI | ❌ |
| Load balancer intégré | ✅ Caddy | ✅ | ✅ | ❌ |
| Preview envs par PR | ✅ | ❌ | ❌ (outils tiers) | ❌ |
| Courbe d'apprentissage | Faible | Moyenne | Haute | Moyenne |

## Stack technique

```
┌─────────────────────────────────────────┐
│ CLI (Bun + TypeScript)                   │
│ ┌───────┐ ┌──────┐ ┌──────┐ ┌────────┐ │
│ │Config │ │Secrets│ │Build │ │Deploy  │ │
│ │YAML   │ │AES   │ │flox  │ │Engine  │ │
│ └───────┘ └──────┘ │OCI   │ └────────┘ │
│                    └──────┘             │
│ ┌──────────────────────────────────────┐│
│ │ SSH Pool (ssh2, parallèle, retry)    ││
│ └──────────────────────────────────────┘│
│ ┌──────────────────────────────────────┐│
│ │ SQLite (state.db local)              ││
│ └──────────────────────────────────────┘│
└────────────────┬────────────────────────┘
                 │ SSH
     ┌───────────┼───────────┐
     ▼           ▼           ▼
┌─────────┐ ┌─────────┐ ┌─────────┐
│Serveur A│ │Serveur B│ │Serveur C│
│ flox    │ │ Docker  │ │ Caddy   │
│ Caddy   │ │ Caddy   │ │ flox    │
│Workload1│ │Workload2│ │Workload1│
└─────────┘ └─────────┘ └─────────┘
```

- **CLI** : Bun + TypeScript — un seul binaire, zéro dépendance
- **Runtime** : flox.dev (environnements Nix reproductibles) ou conteneurs OCI (Docker/Podman)
- **Proxy** : Caddy — reverse proxy, load balancing, SSL automatique
- **State** : SQLite embarqué — déploiements, releases, health logs, locks
- **Secrets** : AES-256-GCM — chiffrés au repos, injectés au déploiement
- **Agent** : Optionnel (TypeScript compilé avec `bun build --compile`), agentless par défaut

## Fonctionnalités principales

- **Déploiement multi-serveur** : SSH parallèle avec limite de concurrence configurable
- **3 stratégies** : Rolling, Blue-Green, Canary
- **3 environnements** : Staging, Test, Production (configurables dans `dnx.yaml`)
- **Build reproductible** : flox.dev (lockfile) ou Docker/OCI (BuildKit sans daemon)
- **Caddy intégré** : Reverse proxy, load balancing (`round_robin`, `least_conn`, `ip_hash`), SSL auto
- **Secrets managés** : `dnx secrets set DATABASE_URL=...` → chiffré AES-256-GCM → injecté au deploy
- **Health checks** : Sondes HTTP/TCP/command, state machine, historique SQLite
- **Rollback** : `dnx rollback web` → retour au tag précédent
- **Lock system** : Verrous SQLite + distants pour éviter les déploiements concurrents
- **Agent optionnel** : Process supervisor, buffer de logs, restart policy
