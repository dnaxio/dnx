# Docker & OCI

DNX supporte les conteneurs OCI (Docker/Podman) en parallèle de flox.dev.

## Configuration

```yaml
workloads:
  - name: web
    driver: docker
    dockerfile: ./Dockerfile
    registry: ghcr.io/org/web
    start_cmd: ""   # Optionnel : override l'entrypoint
    ports:
      - 3000
    env:
      NODE_ENV: "production"
    oci:
      build_args:
        NODE_VERSION: "22"
      secrets:
        - NPM_TOKEN
      volumes:
        - /data/uploads:/app/uploads
```

## Commandes

```bash
# Build l'image OCI
dnx oci build web --tag v1.2.3

# Push vers le registry
dnx oci push web --tag v1.2.3

# Pull sur un serveur
dnx oci pull web prod-1

# Lister les conteneurs
dnx oci images prod-1

# Nettoyer les anciennes images
dnx oci cleanup prod-1 --keep 5
```

## Workflow complet

```bash
# 1. Build
dnx oci build web

# 2. Push
dnx oci push web

# 3. Déployer (le deploy engine détecte le runtime docker)
dnx deploy web --env staging

# 4. Logs
dnx workload logs web --follow
# → docker logs -f dnx-web
```

## BuildKit sans Docker

DNX utilise **BuildKit** (`buildctl`) pour construire les images OCI **sans daemon Docker** :

```bash
# Pas besoin de Docker Desktop !
# BuildKit standalone suffit
brew install buildkit    # macOS
# ou
apt install buildkit     # Linux

dnx oci build web
```

## Podman (rootless)

```yaml
workloads:
  - name: web
    driver: podman    # Au lieu de docker
```

Podman fonctionne sans daemon et sans privilèges root.
