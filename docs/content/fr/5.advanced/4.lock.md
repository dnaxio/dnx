# Lock System

DNX utilise un système de verrous pour empêcher deux déploiements concurrents sur la même ressource.

## Commandes

```bash
# Lister les locks actifs
dnx lock list

# Libérer un lock (force)
dnx lock release deploy:web:production
```

## Types de locks

| Resource | Description |
|----------|-------------|
| `deploy:global` | Un seul déploiement à la fois (global) |
| `deploy:<workload>:<env>` | Un déploiement par workload/environnement |
| `server:<name>` | Verrou au niveau serveur |
| `proxy:reload` | Rechargement Caddy exclusif |

## Fonctionnement

- **Lock local** : SQLite (`locks` table) — empêche deux CLI concurrents sur la même machine
- **Lock distant** : Fichier `/tmp/dnx-lock-<resource>.lock` sur le serveur — empêche deux CLI différentes de déployer sur le même serveur
- **TTL** : Les locks expirent automatiquement après 30 minutes (configurable) pour éviter les locks zombies
- **Nettoyage** : `Bun.cron()` nettoie périodiquement les locks expirés

## Dans le code

```ts
import { withLock } from "dnx";

await withLock("deploy:web:production", async () => {
  await deployWorkload("web", "production");
});
```
